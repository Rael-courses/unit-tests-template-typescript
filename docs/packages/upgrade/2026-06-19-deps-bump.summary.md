## Breaking changes traversés — et pourquoi / comment on les gère

- **eslint 10 supprime le format `.eslintrc`** (flat config seule depuis eslint 9).
  → `.eslintrc` supprimé, remplacé par `eslint.config.mjs` (flat config officielle :
  `@eslint/js` + meta-package `typescript-eslint`). Un script `"lint": "eslint ."` est
  ajouté (avant, eslint était déclaré mais jamais exécuté — config morte).
- **TypeScript 6.0 déprécie `baseUrl`** (`TS5101`). Conformément aux release notes TS 6.0
  (« remove `baseUrl` and add the prefix to their `paths` entries »), `baseUrl` est
  **supprimé** et `paths` devient `"@/*": ["./src/*"]`. Aucun `ignoreDeprecations` n'est
  donc nécessaire. Répercussions :
  - `jest.config.js` ne lit plus `compilerOptions.baseUrl` (`modulePaths: ["<rootDir>"]`).
  - `tsconfig-paths/register` (flow `dev`/nodemon) résout l'alias `@/` **sans** `baseUrl`
    en v4.2.0 — vérifié par un run réel (cf. Validation).
- **TypeScript 6.0 exige un `rootDir` explicite** (`TS5011`) quand `outDir` est défini.
  → `"rootDir": "./src"` ajouté à `tsconfig.build.json` (même layout `dist/` qu'avant).
- **Globals jest introuvables sous TS 6 + @types/jest 30** (`TS2593` dans `jest.setup.ts`).
  → `"types": ["jest", "node"]` ajouté à `tsconfig.json` pour charger explicitement les
  déclarations ambiantes.
- **Conflit `URLPattern` @types/node ↔ lib standard TS 6** (`TS2403`).
  → `"skipLibCheck": true` ajouté (défaut recommandé : ne type-check pas les `.d.ts`
  tiers ; neutralise les conflits inter-`@types`).
- **jest 30** : aucun usage impacté (matchers core `describe/it/expect/toBe`,
  `testEnvironment: "node"`, pas de jsdom).

## À surveiller post-merge

- **TS 7** : non installable tant que `ts-jest` (`<7`) et `typescript-eslint` (`<6.1.0`)
  ne publient pas son support. Quand ce sera le cas, c'est un bump séparé — le retrait de
  `baseUrl` fait ici prépare déjà le terrain.
- `skipLibCheck: true` masque les erreurs dans les `.d.ts` tiers : si un type tiers casse,
  l'erreur ne remontera plus au build (compromis standard et assumé).
- Vérifier en local que `npm run dev` (watch nodemon) démarre bien — le binaire sous-jacent
  a été validé, le wrapper nodemon non (watcher continu).
